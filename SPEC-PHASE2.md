## 一、第二階段要做什麼

讓「上班吃什麼」從查詢工具，變成有同事互動的 App。本階段聚焦三件事：

1. **使用者身分**：員工用公司 Email 登入，App 認得「你是誰」。
2. **揪飯友**：員工可以發起午餐揪團，其他同事報名一起去。
3. **求跑腿**：員工可以發起代買需求，由其他同事接單代買；App 以「付款勾選記帳」協助雙方追蹤誰付了錢。

### 本階段「明確不做」

- **不做點數與排行榜**（移至第三階段）。第二階段先把兩個互動功能本身做穩。
- 不做任何真實金流。求跑腿的金錢由同事私下結清，App 只記帳與提醒，不經手任何錢。
- 不做即時聊天室。
- 不做拍照算熱量（第四階段）。

### 開發順序建議（重要）

雖然本文件一次寫完整，但實際用 Claude Code 開發時，**請務必一個功能一個功能做**：

1. 先做「使用者登入」。
2. 再做「揪飯友」，從頭到尾做完並測試過。
3. 最後才做「求跑腿」。

不要三個一起做，會難以掌控。請 Claude Code 遵守這個順序，每完成一塊先停下讓開發者確認。

## 二、使用者身分與登入

### 登入方式

- 使用**公司 Email 登入**。理由：這是公司內部 App，用公司 Email 可天然確保只有同事能加入，並作為每個人的唯一身分。
- 請 Claude Code 選擇對零基礎開發者最易設定、且安全的登入方案（例如 Email 魔術連結 / 驗證碼，或成熟的第三方登入服務），並一步步引導設定。避免自行從零實作密碼系統。
- 可考慮限定只有特定公司網域（例如 `@公司.com`）的 Email 能註冊，請 Claude Code 評估並建議。

### 使用者資料

| 欄位 | 說明 |
|------|------|
| `id` | 系統自動產生的唯一編號 |
| `email` | 公司 Email |
| `displayName` | 顯示名稱（暱稱，使用者可自訂） |
| `avatar` | 大頭貼（可先用預設圖示，或從名字產生色塊） |
| `department` | 部門（選填，方便揪同部門的人） |
| `createdAt` | 加入時間 |

> 注意：本階段使用者資料**不含點數欄位**，點數待第三階段再加。

第一次登入後，請使用者設定顯示名稱與（選填）部門。

## 三、揪飯友功能

### 核心流程

1. 員工 A 發起一個「飯局」：選一間特約店家、設定出發時間、想揪幾人、附一句話。
2. 飯局出現在「揪團列表」，其他同事看得到。
3. 員工 B、C 看到有興趣，按「報名」。
4. 出發時間到，大家一起去。
5. 飯局結束後，發起人可標記「完成」。

### 資料結構：飯局（meetup）

| 欄位 | 說明 |
|------|------|
| `id` | 飯局唯一編號 |
| `hostId` | 發起人的使用者 id |
| `storeId` | 要去的店家（對應第一階段店家資料）|
| `departTime` | 出發時間 |
| `maxPeople` | 想揪的人數上限（含發起人）|
| `note` | 發起人的一句話，例如「想吃辣的來」|
| `status` | 狀態：招募中 / 已成團 / 已完成 / 已取消 |
| `participantIds` | 已報名的使用者 id 清單 |
| `createdAt` | 建立時間 |

### 畫面

**畫面 A：揪團列表頁**
- 顯示所有「招募中」「已成團」的飯局，依出發時間排序。
- 每張卡片：店家名稱與照片、出發時間、發起人、目前 X/Y 人、note。
- 過了出發時間或已完成的飯局收進「歷史」分頁。
- 明顯的「+ 發起飯局」按鈕。

**畫面 B：發起飯局頁**
- 選店家（從第一階段店家清單挑選）。
- 選出發時間（提供今天午餐常見時段的快速選項）。
- 設定人數上限。
- 填一句話 note。
- 送出後回到列表，發起人自動算第一個參加者。

**畫面 C：飯局詳情頁**
- 店家資訊、出發時間、note。
- 參加者清單（大頭貼與名字）。
- 「報名 / 取消報名」按鈕（發起人看到的是「標記完成 / 取消飯局」）。
- 人數滿了顯示「已成團」，報名按鈕變灰。

### 規則

- 一個人同一午餐時段建議只報名一個飯局，報名第二個時給予溫和提示（不強制禁止）。
- 發起人不可在有人報名後直接刪除飯局，只能「取消」，並讓已報名者看得到狀態變更。
- 出發時間已過的飯局自動轉為「已完成」或「已過期」。

## 四、求跑腿功能

### 設計核心：App 不碰真實金錢

求跑腿牽涉買便當的錢。**本 App 絕不經手任何真實金流**，它的角色是「代買任務看板」加「付款勾選記帳本」：

- 錢由點餐人**直接付現金或轉帳給跑腿者**，在 App 外完成。
- App 只負責記錄「誰已經付錢、誰還沒付」，並讓跑腿者能一鍵通知還沒付的人。
- App 內任何文案都不得暗示 App 會處理付款或代收代付。

### 核心流程

1. 員工 A 發起「跑腿需求」：說明要去哪家店、想買什麼、希望幾點前送到。
2. 其他想搭便車的同事 B、C 在這個需求下「加點」：各自填自己要買的品項與預估金額。
3. 員工 D 願意跑腿，按「我來跑」，成為跑腿者。
4. 跑腿者去買，買回來分給大家。
5. **付款記帳**：跑腿者知道每個人實際該付多少，在 App 裡每個人有一個付款勾選框；收到某人的錢就打勾。
6. 還沒打勾（還沒付錢）的人，跑腿者可以按「提醒付款」，App 發通知給對方。
7. 全部都付清後，這個跑腿需求標記「已結清」。

### 資料結構：跑腿需求（errand）

| 欄位 | 說明 |
|------|------|
| `id` | 跑腿需求唯一編號 |
| `creatorId` | 發起人的使用者 id |
| `storeId` | 要買的店家（對應第一階段店家資料）|
| `deadline` | 希望幾點前送達 |
| `note` | 發起人說明 |
| `runnerId` | 跑腿者的使用者 id（一開始為空，有人接單後填入）|
| `status` | 狀態：徵求跑腿中 / 跑腿者已接單 / 購買中 / 待結清 / 已結清 / 已取消 |
| `createdAt` | 建立時間 |

### 資料結構：跑腿訂單項目（errand_order）

每個參與者在某個跑腿需求下的點單，是一筆 errand_order：

| 欄位 | 說明 |
|------|------|
| `id` | 訂單項目唯一編號 |
| `errandId` | 屬於哪個跑腿需求 |
| `userId` | 點餐人的使用者 id |
| `items` | 想買的品項描述（文字即可，例如「排骨便當不要香菜 x1」）|
| `estimatedAmount` | 點餐人預估金額 |
| `actualAmount` | 實際金額（跑腿者買完後可填正確金額）|
| `isPaid` | 是否已付錢給跑腿者（true / false，就是那個勾選框）|

### 畫面

**畫面 D：跑腿需求列表頁**
- 顯示「徵求跑腿中」「跑腿者已接單」「購買中」的需求。
- 每張卡片：店家、希望送達時間、發起人、目前幾人加點、狀態、有沒有人接單。
- 明顯的「+ 發起跑腿需求」按鈕。

**畫面 E：發起跑腿需求頁**
- 選店家、設定希望送達時間、填說明。
- 送出後發起人可順便填自己要買的品項（自己也是一筆 errand_order）。

**畫面 F：跑腿需求詳情頁**
- 店家、希望送達時間、說明、目前狀態。
- 「我來跑」按鈕（還沒有人接單時，其他人可按）。
- 「加點」按鈕（其他同事按了可填自己的品項與預估金額）。
- 點單清單：列出每個人的品項、金額。
- **付款記帳區（重點）**：
  - 跑腿者接單後，這區出現每位點餐人一行：名字、品項、金額、一個**付款勾選框**。
  - 只有跑腿者可以勾選「已收到這個人的錢」。
  - 每個沒打勾的人旁邊有「提醒付款」按鈕，跑腿者按了會發通知給對方。
  - 全部打勾後，需求可標記「已結清」。
- 點餐人自己看這個頁面時，看得到自己「已付 / 未付」的狀態。

### 規則

- 一個跑腿需求同時只能有一位跑腿者。
- 跑腿者接單後，發起人不能直接刪需求，只能取消（並通知相關人）。
- 金額欄位是給人看的記帳數字，App 不做任何實際扣款或轉帳。
- 「提醒付款」通知的文字保持友善（例如「嘿，記得把午餐錢給 OOO 喔！」）。

## 五、通知功能（本階段最小範圍）

第二階段需要基本通知，支援以下情境：

- 我發起的飯局有人報名 / 我報名的飯局狀態改變。
- 我的跑腿需求有人接單 / 有人加點。
- 我是跑腿者，有點餐人尚未付款（手動觸發的「提醒付款」）。
- 我欠某人午餐錢，收到對方的付款提醒。

實作方式請 Claude Code 建議對零基礎者友善的做法（例如 App 內的通知列表即可，不一定要做手機推播）。先做 App 內通知中心，手機推播留待之後。

## 八、給第三階段的提醒

第三階段預計加入**點數與排行榜**：參與揪飯友、完成跑腿等行為可累積榮譽點數，用於排行榜與成就徽章。

重要原則先記錄於此：點數為**純榮譽性質，不可兌換真實金錢或獎勵**，App 內文案不得暗示分潤。點數規則應集中於設定檔以便調整。是否將點數連結真實獎勵，須待產品成熟、有實際使用數據後，另尋財務與法律專業意見評估。
