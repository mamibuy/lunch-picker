# 上班吃什麼 — 第一階段規格文件（會員系統 + 雙後台版）

> 這份文件是要交給 Claude Code 的開發藍圖。
> 放在專案根目錄，命名為 `SPEC.md`。
> 開發者背景：完全沒寫過程式，請 Claude Code 全程用淺白方式說明，需要操作時明確指示每一步該點哪裡、打什麼。

## 一、產品定位與本階段範圍

「上班吃什麼」是給公司福委會與員工使用的午餐工具。第一階段建立**會員系統與雙層後台**，並完成最基礎的「店家瀏覽」功能。

### 第一階段要做的

1. **員工會員系統**：員工用公司 Email 註冊登入，有個人資料頁。
2. **店家瀏覽**：登入後可看到福委會建立的特約店家列表與詳情。
3. **福委會後台**：福委會幹部能在 App 內管理店家、查看員工名單、停用離職員工、設定其他福委會幹部。
4. **手機優先 + PWA**：手機可加到主畫面，體驗接近 App。

### 第一階段明確不做

- 揪飯友、求跑腿（第二階段）。
- 點數與排行榜（第三階段）。
- 拍照算熱量（第四階段）。
- 不做多公司 SaaS 平台、不做「平台管理員」介面。但資料結構需預留未來擴充為 SaaS 的空間（見「未來擴充考量」一節）。
- 不上架 App Store / Google Play。

### 服務對象

**本階段僅服務一家公司**（你們公司）。公司名稱、公司網域等資訊集中於設定檔，未來變多公司時只需改設定層即可。

## 二、技術選型

| 項目 | 選擇 | 原因 |
|------|------|------|
| 前端框架 | Next.js（React） | 主流、好維護、PWA 支援成熟 |
| 樣式 | Tailwind CSS | 快速做出乾淨的手機版面 |
| PWA | manifest + service worker | 加到主畫面、全螢幕運作 |
| 資料庫與登入 | **Supabase**（推薦） | 對零基礎友善、有免費額度、內建 Email 登入與資料庫；請 Claude Code 一步步引導設定，並避免讓使用者處理複雜金鑰 |
| 部署 | Vercel | 一鍵部署、有免費額度 |

> 如果 Claude Code 認為有更適合零基礎使用者的替代方案，可主動提出，但須先說明利弊讓開發者選擇。

## 三、三種角色與權限

本階段有兩種「實際使用」的角色，第三種預留設計但**不做介面**。

| 角色 | 說明 | 第一階段是否做介面 |
|------|------|------|
| 一般員工 | 公司員工，能瀏覽店家、看自己個資 | 是 |
| 福委會幹部 | 同時是員工，額外能進管理後台 | 是 |
| 平台管理員 | 未來變 SaaS 時的最上層，本階段不做 | 否（資料結構預留欄位） |

### 重要設計原則：角色用「旗標」表示，不要用「分離的使用者表」

福委會幹部本質上也是員工。資料庫設計時，**一個使用者一筆資料**，用一個 `role` 欄位或多個布林旗標（例如 `isStaffCommittee`、`isPlatformAdmin`）來標示額外身份。這樣未來員工升任福委會、或福委會卸任，只需切換旗標，不用搬資料。

### 權限規則

- 一般員工：可看店家、可編輯自己的個資、不能進後台。
- 福委會幹部：除員工權限外，可進「福委會後台」管理店家、員工名單、設定其他幹部。
- 系統內第一位福委會幹部，由開發者（你）在資料庫直接設定，請 Claude Code 帶你操作。之後新增幹部由現有幹部在後台操作。

## 四、員工資料欄位

### 必要欄位（系統運作必須）

| 欄位 | 說明 |
|------|------|
| `id` | 系統產生的唯一編號 |
| `email` | 公司 Email，作為登入帳號，須符合公司網域 |
| `displayName` | 顯示名稱，第一次登入時設定 |
| `department` | 部門，第一次登入時設定 |
| `isActive` | 是否啟用（離職員工由福委會停用，不刪除以保留歷史紀錄）|
| `isStaffCommittee` | 是否為福委會幹部 |
| `isPlatformAdmin` | 是否為平台管理員（本階段都是 false，預留欄位）|
| `createdAt` | 加入時間 |

### 選填欄位（員工自願提供，個資保護）

| 欄位 | 用途說明（會顯示給員工看）|
|------|------|
| `phone` | 電話，用於緊急聯絡或求跑腿確認；**選填** |
| `commonAddress` | 常用地址，用於未來求跑腿外送；**選填** |
| `avatar` | 大頭貼，未提供時系統用名字產生色塊；**選填** |
| `birthday` | 生日（只取月日，不取年），未來可用於生日祝賀；**選填** |

### 個資保護要求（給 Claude Code）

- 個人資料頁的選填欄位旁邊必須明確說明「用途」與「選填」標記。
- 任何選填欄位都不能阻擋使用者使用 App 主功能。
- 個資頁要有清楚的「儲存」按鈕，不要自動同步。
- 別的員工瀏覽某人時，**不可看到電話與常用地址**，只能看到 displayName、avatar、department。
- 福委會幹部在後台**也不可任意查看員工電話與常用地址**，除非有正當管理需求且需明確標示「正在檢視個資」。

> 設計理由：電話、常用地址屬於個人敏感資料，收集與使用都應最小化。

## 五、員工端畫面

### 畫面 1：登入頁
- 標題「上班吃什麼」與一句俏皮標語。
- Email 輸入框，按「寄送登入連結」按鈕。
- 系統寄出 Email 魔術連結（或驗證碼，由 Claude Code 選擇實作上最穩定的方式）。
- 限制：僅允許公司網域的 Email（網域集中於設定檔，方便日後修改）。

### 畫面 2：首次登入引導頁
- 填寫顯示名稱（必填）、部門（必填）。
- 提示「以下可以稍後再填」：電話、常用地址、生日、大頭貼。
- 跳過或填完都進入店家列表頁。

### 畫面 3：店家列表頁（登入後首頁）
- 上方：App 標題、自己的大頭貼（點開進個人頁）、福委會幹部會多一個「後台」入口。
- 分類篩選列：全部 / 便當 / 麵食 / 飲料 / 早午餐 / 異國料理 / 其他。
- 店家卡片清單：照片、店名、分類、價位、步行分鐘、**特約內容（顯眼）**、一句話介紹。
- 預設依步行分鐘由近到遠。
- 只顯示「啟用中」的店家。

### 畫面 4：店家詳情頁
- 大張照片、店名、分類、價位。
- **特約內容（最顯眼區塊）**。
- 介紹文字、地址（點開 Google Maps）、電話（點了撥號）、標籤。
- 返回按鈕。

### 畫面 5：個人資料頁
- 顯示與編輯：displayName、department（必填）。
- 選填區（含用途說明）：phone、commonAddress、birthday、avatar。
- 「登出」按鈕。

## 六、店家資料結構

| 欄位 | 必填 | 說明 |
|------|------|------|
| `id` | 是 | 系統產生 |
| `name` | 是 | 店名 |
| `category` | 是 | 便當 / 麵食 / 飲料 / 早午餐 / 異國料理 / 其他 |
| `description` | 否 | 一句話介紹 |
| `address` | 是 | 地址 |
| `mapUrl` | 否 | Google Maps 連結 |
| `phone` | 否 | 店家電話 |
| `priceRange` | 否 | $ / $$ / $$$ |
| `discount` | 是 | 特約內容 |
| `walkMinutes` | 否 | 步行分鐘（數字）|
| `photoUrl` | 否 | 店家照片 |
| `tags` | 否 | 標籤陣列 |
| `isActive` | 是 | 是否啟用 |
| `createdAt`, `updatedAt` | 是 | 系統自動 |

請 Claude Code 在開發階段先建立 5～8 筆假資料，方便測試。

## 七、福委會後台

進入方式：福委會幹部登入後，首頁多出「後台」入口；一般員工看不到此入口。

### 後台首頁
- 三大區塊入口：店家管理、員工管理、後台設定。
- 顯示基本統計：目前啟用店家數、目前員工數。

### 後台子畫面 A：店家管理
- 列表：顯示所有店家（含未啟用的），可搜尋、可篩選分類。
- 每一列：店名、分類、特約內容、是否啟用、編輯、停用/啟用。
- 「新增店家」按鈕：開啟表單，填上述店家欄位後儲存。
- 編輯店家：所有欄位皆可改。
- 停用而非刪除：停用後員工端看不到，但歷史紀錄保留。

### 後台子畫面 B：員工管理
- 列表：顯示所有員工，可搜尋、可篩選部門。
- 每一列顯示：displayName、email、department、是否為福委會幹部、是否啟用。
- **不顯示電話、常用地址**。
- 動作：
  - 停用/啟用員工（例如離職）。
  - 切換是否為福委會幹部。
  - 「檢視個資」（會在記錄中留下查看紀錄，並提示「正在檢視同事的私人個資」）。
- 員工是由員工自行 Email 註冊產生的，福委會不需要也不能手動建立員工帳號。

### 後台子畫面 C：後台設定
- 顯示「公司網域」設定（讀取設定檔的內容，本階段為唯讀）。
- 顯示分類列表（如要新增分類，本階段請改設定檔；未來可做成可編輯）。

### 後台設計要求
- 後台介面與員工端**視覺上明顯不同**（例如配色較沉穩），讓福委會清楚自己正在管理介面。
- 後台所有寫入動作（新增、編輯、停用）都要有確認步驟，避免誤觸。
- 任何破壞性動作（如停用員工）操作後可在「後台操作記錄」頁查到誰在何時做了什麼，方便事後追查。

## 八、設計要求

- **手機優先**：員工端版面以手機直式為主（375～430px），桌機開不跑版。
- **後台**：手機與桌機都要能用，但福委會幹部多半會在桌機操作後台，桌機體驗也要顧。
- **特約資訊要突出**：員工最在意「這家有沒有特價」，每個畫面都要一眼看到。
- **風格**：員工端明亮活潑（暖色系）；後台沉穩專業。
- **PWA**：App 圖示、主題色、加到主畫面後全螢幕。

## 九、未來擴充考量（重要）

本階段雖然只服務一家公司，但要為「未來變多公司 SaaS」預留設計：

1. **公司資料抽象化**：公司名稱、公司網域、品牌色等不寫死在程式中，集中於設定檔或資料表（即使只有一筆資料）。
2. **使用者表預留 `companyId` 欄位**：本階段都填同一個值，但欄位存在，未來變多公司時不用大規模改資料。
3. **店家表也預留 `companyId` 欄位**：同上理由。
4. **`isPlatformAdmin` 欄位**：本階段全為 false，預留給未來平台管理員。

請 Claude Code 在資料庫設計時主動納入這些欄位，並在程式碼中以「目前固定為單一公司」的方式運作，但結構支援未來擴充。

> 提醒：這只是「資料結構預留」，不是「現在做 SaaS」。本階段一律不做多公司介面、不做平台管理員介面、不做訂閱付費等 SaaS 功能。

## 十、開發順序（重要）

對零基礎開發者，**嚴格分段**，每段做完先停下確認再做下一段：

1. **第 1 段：環境建立**。建立 Next.js 專案、設定 Tailwind、設定 Supabase 連線。
2. **第 2 段：登入與會員系統**。Email 魔術連結登入、限制公司網域、首次登入引導頁、個人資料頁。
3. **第 3 段：店家列表與詳情**。先用假資料展示，畫面做出來再串資料庫。
4. **第 4 段：福委會後台 — 店家管理**。新增、編輯、停用店家。
5. **第 5 段：福委會後台 — 員工管理**。查看名單、停用、切換幹部身份。
6. **第 6 段：個資保護細節**。隱藏私密欄位、檢視紀錄、確認步驟。
7. **第 7 段：PWA 設定**。manifest、service worker、圖示。
8. **第 8 段：部署上線**。Vercel 部署、測試完整流程。

每段結束後請 Claude Code 主動提示「這段完成了，請開發者測試以下幾點……」並等待確認再動下一段。
