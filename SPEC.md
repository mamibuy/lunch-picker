# 上班吃什麼 — 第一階段規格文件（店家網 / PWA 版）

> 這份文件是要交給 Claude Code 的開發藍圖。
> 把它放在專案根目錄，命名為 `SPEC.md`，Claude Code 開發時會持續參考它。
> 開發者背景：完全沒寫過程式，請 Claude Code 用淺白的方式說明每一步，並主動處理技術細節。

---

## 一、這個階段要做什麼

第一階段只做一件事：**讓員工能在手機上瀏覽福委會特約店家**。

不做：使用者帳號、揪飯友、求跑腿、點數、拍照算熱量。這些是後面階段的事，這階段刻意不碰，才能快速做出能用的東西。

成功標準：福委會同事打開網址，就能看到所有特約店家、知道有什麼特價、知道怎麼去；而且可以把網頁「加到主畫面」，像 App 一樣使用。

---

## 二、產品定位

- **形式**：手機網頁版，並做成 **PWA（漸進式網頁應用）**——員工用手機瀏覽器打開後可「加到主畫面」，桌面出現圖示，點開全螢幕運作，體驗接近原生 App。**不上架 App Store / Google Play。**
- **使用者**：公司內部員工，中午想找地方吃飯。
- **使用情境**：午休前 10 分鐘，站在辦公室，單手滑手機決定中午吃哪家。
- **設計基調**：輕鬆、有趣、好點。不要做成嚴肅的企業系統。

---

## 三、技術選型（給 Claude Code 的建議）

| 項目 | 選擇 | 原因 |
|------|------|------|
| 前端框架 | Next.js（React） | 主流、好維護、之後加功能容易，PWA 支援成熟 |
| 樣式 | Tailwind CSS | 快速做出乾淨的手機版面 |
| PWA | 加上 manifest 與 service worker | 讓網頁能「加到主畫面」、有 App 圖示與啟動畫面 |
| 資料來源 | **Google Sheet** | 福委會在試算表填資料即可維護店家，不需開發後台 |
| 部署 | Vercel | 一鍵部署、有免費額度 |

---

## 四、店家資料：用 Google Sheet 當後台

### 運作方式

福委會維護一個 Google 試算表，一列代表一間店家。App 啟動時讀取這張試算表的內容並顯示。福委會要新增 / 修改 / 停用店家，只要在試算表上編輯，不需要任何後台系統，也不需要找開發者。

### 請 Claude Code 協助處理的事

1. 設計試算表的欄位（見下方），並產生一份「給福委會的填寫說明」。
2. 實作從 Google Sheet 讀取資料的程式。建議用「將試算表發布為 CSV」或 Google Sheets API 的方式，請 Claude Code 選擇對零基礎使用者最不容易出錯、且不需要管理金鑰的做法，並在過程中一步步引導設定。
3. 把「讀取店家資料」獨立成單一函式，未來要換資料來源時只需改這裡。
4. 做基本的容錯：某一列資料缺漏或格式錯誤時，跳過該列並在主控台提示，不要讓整個 App 掛掉。

### 試算表欄位

| 欄位（標題列） | 必填 | 說明 |
|----------------|------|------|
| 店名 | 是 | 店家名稱 |
| 分類 | 是 | 便當 / 麵食 / 飲料 / 早午餐 / 異國料理 / 其他 |
| 介紹 | 否 | 一句話介紹 |
| 地址 | 是 | 店家地址 |
| 地圖連結 | 否 | Google Maps 連結 |
| 電話 | 否 | 店家電話 |
| 價位 | 否 | $ / $$ / $$$ |
| 特約內容 | 是 | 例如「憑員工證 9 折」「滿 100 折 15」 |
| 步行分鐘 | 否 | 從公司步行幾分鐘（數字） |
| 照片連結 | 否 | 店家照片網址 |
| 標籤 | 否 | 以頓號分隔，例如「有素食、可外送、份量大」 |
| 是否顯示 | 是 | 「是」才會顯示在 App，「否」可暫時下架某店家 |

> 請 Claude Code 先建立一份含 5～8 筆假資料的範例試算表內容（或本機 JSON），方便開發階段測試，不必等福委會真的填好才能開發。

## 五、畫面規格

### 畫面 1：店家列表頁（首頁）

打開網址第一個看到的畫面。

- 最上方：App 標題「上班吃什麼」+ 一句俏皮標語（例如「午休煩惱終結者」）。
- 標題下方：分類篩選列，橫向可滑動的標籤按鈕：全部 / 便當 / 麵食 / 飲料 / 早午餐 / 異國料理 / 其他。點了就只顯示該分類。
- 主體：店家卡片清單，每張卡片顯示：
  - 店家照片（沒照片時顯示分類對應的預設圖示）
  - 店名
  - 分類標籤、價位、步行分鐘數
  - **特約內容**（重點，要明顯，用顯眼顏色或加上「特約」標記）
  - 一句話介紹
- 點任一張卡片 → 進入「店家詳情頁」。
- 排序：預設依步行分鐘數由近到遠。
- 只顯示試算表中「是否顯示」為「是」的店家。

### 畫面 2：店家詳情頁

- 大張店家照片。
- 店名、分類、價位。
- **特約內容**（最顯眼的區塊）。
- 介紹文字。
- 地址（點了開 Google Maps）。
- 電話（點了直接撥號）。
- 標籤。
- 返回按鈕回到列表頁。

### 畫面 3（加分，有時間再做）：今天吃這家！

- 列表頁上方放一顆按鈕「幫我決定」。
- 點下去，從目前篩選的店家中隨機跳出一間，給個有趣的動畫或文案。
- 這是為第二階段的「趣味互動」鋪路的最小版本。

## 六、設計要求

- **手機優先**：版面以手機直式螢幕為主（寬度約 375～430px），桌機開也不能跑版。
- **好點**：按鈕和卡片夠大，單手好操作。
- **特約資訊要突出**：使用者最在意「這家有沒有特價」，這個資訊在每個畫面都要一眼看到。
- **載入要快**：圖片要做延遲載入。
- **風格**：明亮、活潑、有食慾感的配色（暖色系），不要灰灰的企業風。
- **PWA 細節**：要有 App 圖示、適當的主題色、加到主畫面後以全螢幕開啟。

---

## 七、明確不做的事（避免範圍擴張）

- 不做使用者註冊 / 登入。
- 不做揪飯友、求跑腿、點數系統。
- 不做拍照上傳、熱量計算。
- 不做自製後台管理介面（店家資料一律由 Google Sheet 維護）。
- 不做評論 / 評分功能。
- 不上架 App Store / Google Play。

以上都是後續階段的事，這階段做了反而拖慢進度。

- [ ] 手機打開網址，看到店家列表。

- [ ] 可以用分類篩選店家。
- [ ] 每間店都看得到特約內容。
- [ ] 點店家可以看詳情。
- [ ] 詳情頁可以點地址開地圖、點電話撥號。
- [ ] 「是否顯示」為「否」的店家不會出現。
- [ ] 桌機開也不跑版。
- [ ] 可以在手機上「加到主畫面」，出現 App 圖示，點開全螢幕運作。
- [ ] 已部署到 Vercel，有一個可分享的網址。
- [ ] 有一份給福委會的「試算表填寫說明」。
